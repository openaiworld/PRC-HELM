# 命名与方法论来源

本文件以压缩形式保留 PRC-HELM 的命名含义与“理论到工程”的映射。仅当用户询问 skill 设计原因、完整思想映射，或要求修订方法论本身时读取。

## 命名含义

**PRC-HELM** 的 HELM 展开为 **Historical-Engineering Liberation Methodology**。

| 层次 | 含义 |
|---|---|
| 政治/历史意象 | “舵手”对应“掌舵前进”的中文语境，表达方向感和组织性。 |
| 英文展开 | Historical-Engineering Liberation Methodology，强调从历史经验中抽象工程方法。 |
| 工程彩蛋 | Kubernetes 的包管理器 Helm 已被工程社区熟悉，天然关联“管理、编排、交付”。 |
| 项目含义 | Helm = 掌舵，Agent 作为驾驶项目前进的执行框架。 |

## 映射表

| 来源概念 | 工程解释 | 操作规则 |
|---|---|---|
| 矛盾分析 | 区分主要阻塞与次要问题 | 每轮迭代命名一个主要矛盾；次要矛盾最多三项 |
| 实践论 | 认识通过原型、反思、实现不断升级 | 不确定时做 spike/PoC，并记录结果 |
| 用户路线 | 从用户需求出发，再回到用户验证 | 要求用户故事、验收标准和反馈闭环 |
| 民主集中 | 充分讨论，明确决策，一致执行 | RFC 负责讨论，ADR 记录已接受决策 |
| 灵活战术 | 根据约束动态调整范围与策略 | 压力下砍低优先级范围，保核心交付物 |
| 务实主义 | 结果比形式、流行或口号更重要 | 使用证据化技术选型矩阵 |
| 三维质量 + RMST | 平衡生产力、可靠性、可维护性、可扩展性、可溯源性、工程文化和用户价值 | 用可量化维度评分迭代健康度 |
| 多 Agent 协同 | 把复杂治理分解到指挥、计划、研究、架构、编码、审查、哨兵、史官角色 | 使用 `.ai_helm/state/queue.json` 和 `.ai_helm/state/progress.md` 驱动 Loop |
| 批评与自我批评 | 通过暴露风险和限制来改进 | PR 自审、事故复盘、迭代 retrospective |
| 传帮带 / 知识传承 | 降低人员风险和技能方差 | 在相关场景维护 onboarding、pairing、知识笔记 |
| 失败知识库 | 失败实验可以避免重复浪费 | 将失败 PoC 和被否方案保存在 `.ai_helm/failure-kb/` 或项目既有 failure-kb 中 |

## 护栏

- 将原始理论术语视为工程记忆锚点，而不是在工程输出中做政治宣传。
- 在专业协作场景中优先使用中性工程语言。
- 不要编造历史引文或证据；如果引文重要，先从可靠来源核验。
- 如果某条规则与用户明确指令或仓库规范冲突，应说明冲突，并遵循更高优先级指令。
