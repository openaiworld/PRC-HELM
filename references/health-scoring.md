# 迭代健康度评分

在迭代收尾、编写 retrospective 或审计项目健康度时使用。只给可观察事实打分。证据缺失时标记为未知，并建议下一次如何补充度量。

## 使用步骤

1. 确认本轮主要矛盾是否关闭。
2. 汇总证据：提交、测试、CI、文档、需求、PR、事故、用户反馈。
3. 按四个维度打分；没有证据不默认满分。
4. 输出总分、等级、风险、后续行动。
5. 建议把报告写入 `docs/retrospective/YYYY-MM-DD.md`，并更新 `progress.md`。

## 报告格式

```markdown
# 迭代健康度报告

迭代：<名称或日期>
已关闭的主要矛盾：
下一阶段主要矛盾：

## 先进生产力维度（35 分）

- 核心性能目标达成：__/10
- 核心逻辑有有效测试：__/10
- CI/build/test 流程绿色或已明确验证：__/10
- 技术债可控且已追踪：__/5

## 工程文化维度（25 分）

- 新增或变更行为已有文档：__/10
- 必要的架构/技术决策已有记录：__/10
- Review 或自审有实质内容：__/5

## 用户价值维度（25 分）

- 至少改善一个用户/业务问题：__/10
- 验收标准、UAT 或人工验证已记录：__/10
- 无严重未解决事故；若发生事故则有复盘：__/5

## 团队能力与知识沉淀维度（15 分）

- 未来低上下文维护者也能理解本次工作：__/5
- 有价值的失败尝试或被否方案已保留：__/5
- 相关知识分享、入门或交接说明已更新：__/5

总分：__/100
等级：
- 90-100：优秀，运行基础扎实
- 75-89：良好，可持续
- 60-74：合格，但需要改进
- <60：高风险，需要安排纠偏复盘

## 证据

## 风险

## 建议后续行动
```

## 打分原则

- 保守打分。没有证据的声明不要给满分。
- 小任务不适用的维度不要强行扣分；标记为不适用并说明原因。
- 如果本轮没有代码变更，将代码相关检查替换为等价的产物质量检查。
- 如果用户要求简洁收尾，只输出总分摘要和前三项后续建议。

## 快速评分版

当用户只说“总结一下”且任务较小，可使用简版：

```markdown
主要矛盾：
完成情况：
验证情况：
健康度：__/100（等级）
前三项后续建议：
1.
2.
3.
```

## 证据缺口提示

如果无法评分，明确指出缺口，例如：

- 缺少测试输出，无法判断 CI/build/test 维度。
- 缺少需求或验收记录，用户价值维度只能低置信评分。
- 没有 PR/review 过程，工程文化维度只能依据自审说明评分。
